===============================================================================
4. ANALYSIS
===============================================================================

All 20 edge case vectors + 3 sanity checks produce IDENTICAL results
(same accept/reject decision AND same error code) between the reference
(5x51-bit fiat-crypto) and AVX512 (6x43-bit r43x6) implementations.

Key observations:
- Both implementations correctly reject all small-order public keys
- Both implementations correctly reject S >= L (scalar validation)
- Both implementations correctly reject non-canonical R points
- Sign-then-verify round-trip works identically on both paths
- The fd_ed25519_verify shared code (fd_ed25519_user.c) provides a
  single code path that calls backend-specific primitives:
    - fd_ed25519_point_frombytes_2x (point decompression)
    - fd_ed25519_affine_is_small_order
    - fd_ed25519_double_scalar_mul_base
    - fd_ed25519_point_eq_z1 (final R comparison)
  The pre-checks (scalar validation, small-order rejection) are shared
  code, reducing the divergence surface.

===============================================================================
5. CONCLUSION
===============================================================================

H6 hypothesis is NOT confirmed with these 20 test vectors.

The shared verify logic in fd_ed25519_user.c performs scalar validation
and small-order checks BEFORE reaching the backend-divergent code paths.
This means many edge cases are caught by identical shared code regardless
of the field arithmetic backend.

The remaining attack surface for ref/avx512 divergence is narrow:
- fd_ed25519_point_frombytes_2x (point decompression)
- fd_ed25519_double_scalar_mul_base (the core scalar multiplication)
- fd_ed25519_point_eq_z1 (final comparison)

To more thoroughly test these, one would need:
1. Valid signatures with crafted inputs that stress field boundary
   conditions during scalar multiplication (requires knowing discrete
   logs of specific points)
2. Fuzzing fd_f25519 operations directly between ref and avx512
   (Firedancer already has fuzz_ed25519_sigverify_diff.c for this)
3. Testing the r43x6 representation transitions (unpacked->unsigned->
   nearly_reduced->reduced) with values near carry boundaries
