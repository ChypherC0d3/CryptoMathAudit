3. CRYPTOGRAPHIC ARCHITECTURE
================================================================================

3.1 HIGH-LEVEL ARCHITECTURE (V2)
----------------------------------

LayerZero V2 has four core components:
  1. Immutable Endpoint     - Censorship-resistant message passing
  2. MessageLib Registry    - Append-only registry of verification modules
  3. DVNs                   - Decentralized Verifier Networks
  4. Executors              - Execute feature logic in isolation

Message flow:
  Source Chain:  OApp -> EndpointV2 -> SendUln302 -> [DVN fee paid]
  Off-chain:    DVN detects PacketSent event, verifies payload hash
  Dest Chain:   DVN calls ReceiveUln302.verify() -> commitVerification() ->
                EndpointV2 -> OApp.lzReceive()

3.2 DVN (DECENTRALIZED VERIFIER NETWORK) ARCHITECTURE
-------------------------------------------------------

DVN = smart contract + off-chain infrastructure

Verification Model: X-of-Y-of-N quorum
  - X = required (mandatory) DVNs that ALL must verify
  - Y = threshold of optional DVNs that must verify
  - N = total optional DVN pool

UlnConfig struct:
  - confirmations: required block confirmations
  - requiredDVNCount (X): all must verify
  - optionalDVNCount (N): total optional DVNs
  - optionalDVNThreshold (Y): minimum optional verifications
  - requiredDVNs[]: addresses
  - optionalDVNs[]: addresses

Maximum: 254 DVNs composable per Security Stack

DVN is protocol-agnostic -- can use ANY verification method:
  - ECDSA multi-sig (most common, used by LayerZero Labs DVN)
  - Zero-knowledge proofs
  - Sidechain consensus
  - Native bridges
  - Restaking/slashing (EigenLayer AVS)
  - MPC/threshold signatures

3.3 SIGNATURE SCHEME -- MultiSig.sol (THE KEY CONTRACT)
---------------------------------------------------------

Location: packages/layerzero-v2/evm/messagelib/contracts/uln/dvn/MultiSig.sol

Scheme: ECDSA multi-signature with quorum threshold

Implementation Details:
  - Uses OpenZeppelin's ECDSA.tryRecover()
  - Ethereum signed message prefix: "\x19Ethereum Signed Message:\n32"
  - Each signature is exactly 65 bytes (r=32, s=32, v=1)
  - Signature length validation: requires exactly quorum * 65 bytes
  - Signer ordering: each recovered signer must be lexicographically greater
    than the previous (prevents duplicate signatures)
  - Each recovered signer must be in the authorized signer set
    (EnumerableSet)

verifySignatures(bytes32 _hash, bytes calldata _signatures):
  1. Check _signatures.length == quorum * 65
  2. For each 65-byte chunk:
     a. Compute ethSignedHash = keccak256("\x19Ethereum Signed Message:\n32" || _hash)
     b. ECDSA.tryRecover(ethSignedHash, signature_chunk)
     c. Verify recovered address > previous recovered address (ascending order)
     d. Verify recovered address is an authorized signer
  3. If all pass, signatures are valid

Quorum Safeguards:
  - Quorum cannot be zero
  - Signer count must always exceed quorum
  - Cannot remove signers below quorum threshold

3.4 ULN VERIFICATION FLOW (ReceiveUln302)
-------------------------------------------

ReceiveUln302.verify(bytes calldata _packetHeader, bytes32 _payloadHash,
                     uint64 _confirmations):
  - Called by DVN contracts (msg.sender = DVN address)
  - Stores: hashLookup[headerHash][payloadHash][dvnAddress] = Verification(true, confirmations)
  - Emits PayloadVerified event
  - NOTE: The ULN itself does NOT verify ECDSA signatures -- it only tracks
    which DVN contracts have called verify(). Signature verification happens
    WITHIN each DVN contract (MultiSig.sol).

ReceiveUln302.commitVerification(bytes calldata _packetHeader, bytes32 _payloadHash):
  - Can be called by anyone
  - Checks all required DVNs have verified
  - Checks optionalDVNThreshold met
  - Commits to Endpoint

3.5 DVN.sol EXECUTION AND REPLAY PROTECTION
---------------------------------------------

DVN.sol inherits MultiSig.sol

execute(ExecuteParam[] calldata _params):
  - Admin-only function
  - For each param: hash = hashCallData(vid, target, expiration, callData)
  - Checks hash not in usedHashes (replay protection)
  - Verifies quorum signatures via verifySignatures(hash, signatures)
  - Marks hash as used
  - Executes calldata against target
  - If execution reverts, unmarks hash to allow retry

hashCallData: keccak256(abi.encodePacked(_vid, _target, _expiration, _callData))
  - _vid: unique DVN instance identifier (prevents cross-DVN replay)
  - _target: contract to call
  - _expiration: timestamp-based expiry
  - _callData: the actual function call

OPTIMIZATION NOTE: "to save gas, we don't check hash for some functions
(where replaying won't change the state)" -- exempts stateless operations
like verify() calls via _shouldCheckHash(bytes4 selector).

3.6 NONCE AND REPLAY PROTECTION (Protocol Level)
--------------------------------------------------

Packet structure includes:
  - uint64 nonce: sequential per-channel, prevents replay
  - GUID: globally unique identifier

Exactly-once delivery:
  - Verified hash deleted from lossless channel after delivery
  - Nonces <= lazy inbound nonce are disallowed for re-verification
  - Channel isolation: per-pathway, per-OApp

Lazy nonce mechanism:
  - Tracks highest consecutively delivered message
  - Messages can be verified out of order
  - Execution is sequential from lazy nonce

  - Occurs AFTER Oracle and Relayer have signed off
  - Exploits "drag-along" mechanism in upgrade system

  - Default library pointer can be changed by team for default-configured apps
  - Most projects were using defaults at the time
  - Recommended apps set their own config rather than relying on defaults

Status: V2 architecture redesigned to address these concerns via immutable
Endpoint and append-only MessageLib registry. However, similar trust
assumptions may exist in different forms.

6. ATTACK SURFACE ANALYSIS
================================================================================

6.1 SIGNATURE MALLEABILITY
----------------------------

Status: MITIGATED by OpenZeppelin ECDSA.tryRecover()

OpenZeppelin's ECDSA library protects against EIP-2 signature malleability by:
  - Enforcing s value in lower half of secp256k1 order
  - Validating v is exactly 27 or 28
  - Using tryRecover (returns error instead of reverting on failure)

Residual risk: If any DVN adapter or custom implementation uses raw ecrecover()
without OpenZeppelin, malleability could allow signature reuse.

AUDIT FOCUS: Check DVN adapter contracts in
packages/layerzero-v2/evm/messagelib/contracts/uln/dvn/adapters/
for raw ecrecover usage.

6.2 ecrecover RETURNING address(0)
-------------------------------------

Status: MITIGATED by ECDSA.tryRecover()

ECDSA.tryRecover returns RecoverError.InvalidSignature if recovery yields
address(0). The MultiSig contract uses tryRecover and checks for errors.

Residual risk: Any contract that uses raw ecrecover without checking for
address(0) return. If address(0) were somehow in the signer set, it could
validate invalid signatures.

AUDIT FOCUS:
  - Verify address(0) is never added to signer set
  - Check _setSigner() validates _signer != address(0)
  - Check all DVN adapters for raw ecrecover usage

6.3 MISSING LENGTH CHECKS ON SIGNATURE BYTES
----------------------------------------------

Status: CHECKED in MultiSig.sol

MultiSig.verifySignatures requires: _signatures.length == quorum * 65
This prevents:
  - Undersized signature arrays (fewer sigs than quorum)
  - Oversized arrays (extra data appended)
  - Misaligned parsing

Residual risk: If quorum is 0 (should be prevented by _setQuorum), length
check passes with empty signatures.

AUDIT FOCUS:
  - Verify quorum > 0 is enforced at all paths
  - Check if quorum can be set to 0 via any admin pathway
  - Integer overflow: quorum * 65 for very large quorum values (uint64)

6.4 REPLAY ATTACKS ACROSS CHAINS
----------------------------------

Status: PARTIALLY MITIGATED

Protocol level:
  - Nonce per channel (source/dest pair) prevents protocol-level replay
  - GUID provides global uniqueness
  - Exactly-once delivery with hash deletion after execution

DVN level:
  - hashCallData includes vid (DVN instance ID) -- prevents cross-DVN replay
  - hashCallData includes target address
  - usedHashes mapping prevents same-DVN replay
  - Expiration timestamp provides time-bound validity

RESIDUAL RISKS:
  a) Cross-chain DVN replay: If vid is the same across chains for the same
     DVN deployment, a signed execution on chain A could potentially be
     replayed on chain B if the target address exists on both chains.
     INVESTIGATE: Is vid unique per chain or per DVN instance?

  b) verify() calls exempt from hash checking: DVN.sol optimization skips
     usedHashes for "stateless" operations. Verify that verify() is truly
     idempotent and that replaying it cannot cause harm.
     "to save gas, we don't check hash for some functions"

  c) Application-layer replay: LayerZero protocol guarantees exactly-once
     delivery, but apps that process messages outside standard lzReceive
     flow may lack replay protection. (Sherlock audit found this in a
     lending protocol built on LayerZero.)

6.5 NONCE HANDLING IN MULTI-SIG VERIFICATION
----------------------------------------------

The MultiSig.sol does NOT use nonces. Instead it uses:
  - Hash-based replay protection (usedHashes mapping in DVN.sol)
  - Expiration timestamps
  - Ascending signer address ordering (prevents duplicate sigs, not replay)

Potential issues:
  a) No nonce means ordering of operations is not enforced within the
     expiration window. Two valid signed operations could be executed in
     any order.

  b) If an execution reverts, the hash is UNMARKED (allowing retry).
     This is intentional but creates a window where a previously-failed
     transaction could be retried by anyone with ADMIN_ROLE after
     conditions change.

  c) Expiration check: verify the expiration comparison is strict
     (block.timestamp < expiration, not <=). Off-by-one in expiration
     could extend execution window.

6.6 ADDITIONAL ATTACK VECTORS
-------------------------------

a) DVN Collusion / Compromise:
   - If all required DVNs for an OApp are compromised, arbitrary messages
     can be verified. This is by design (security = union of DVN trust).
   - Investigate: Can a compromised DVN call verify() for packets that
     were never sent? (Answer: Yes, if the DVN contract calls
     ReceiveUln302.verify with arbitrary data)
   - Mitigation: nilify() function can nullify verified packets

b) commitVerification permissionless call:
   - Anyone can call commitVerification once DVN quorum is met
   - Could this be front-run or used to commit at an unfavorable time?

c) Role escalation in DVN.sol:
   - quorumChangeAdmin requires only MultiSig quorum (not full execute flow)
   - If quorum is compromised, admin can be changed
   - Admin can then do: setDstConfig, withdrawFees, execute, grantRole
   - Could lead to DVN takeover

d) Fee manipulation:
   - DVNFeeLib computes fees. If fee calculation is manipulable, DVN
     could be drained or messages could be verified for free.

e) TON/Solana/Aptos specific:
   - Different runtime environments may have different signature
     verification characteristics
   - TON Controller/ULNManager/DVNProxy are in scope but less audited

f) MessageLib upgrade path:
   - V2 uses append-only registry (unlike V1's replaceable default)
   - But: What happens if a new malicious MessageLib is registered?
   - OApps must explicitly opt-in, but verify the opt-in mechanism

g) Lazy nonce gap exploitation:
   - Messages can be verified out of order but execute sequentially
   - Can a blocked message at nonce N prevent all subsequent messages?
   - This is acknowledged as a design feature (lossless delivery)
   - But: Could an attacker intentionally block a nonce to freeze a channel?
