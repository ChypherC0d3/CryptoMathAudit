## 2. EIP Specification Check (CONFIRMED deviation)

EIP-4844 specifies BLOBHASH (0x49) should:
- Read an index from the stack
- Return tx.blob_versioned_hashes[index] if index < len(hashes)
- Return zeroed bytes32 only if index is out of bounds

EIP-7516 specifies BLOBBASEFEE (0x4A) should:
- Push the current block's blob base fee onto the stack
- This is the result of get_blob_gasprice(header)

The implementation deviates from both specifications.

## 6. Nuanced Assessment

There IS a subtle argument to be made: the documentation says these
opcodes are "unsupported," which typically implies they would revert or
be rejected. Instead, the interpreter:
- Accepts the opcodes without error
- Returns plausible but incorrect values (0 and 1)
- Does not revert or signal an error

This "silent failure" behavior could be considered worse than truly
unsupported (which would revert). A contract checking BLOBBASEFEE would
get 1 instead of a revert, potentially making incorrect decisions.

However, at this commit, ZKsync does not support blob transactions on L2
at all, so there are no real blob hashes or blob base fees to return.
The stub values are placeholders for a feature not yet enabled.

This blob opcode behavior could potentially be that intentional bug,
