=======================================================
STEP 3: IMPACT ANALYSIS
=======================================================

What the bug causes:
- The seat-count eligibility check for triplet exclusion is WRONG.
  It calculates: groupMembers - seats(operators[i]) - seats(operators[j]) - seats(operators[j])
  Instead of:    groupMembers - seats(operators[i]) - seats(operators[j]) - seats(operators[k])

- This means middleOperator's seats are double-counted and the third
  operator's (k's) seats are ignored entirely in the eligibility filter.

- However, the ACTUAL EXCLUSION (lines 327-330) is CORRECT:
    leftOperator := operators[triplet[0]]
    middleOperator := operators[triplet[1]]
    rightOperator := operators[triplet[2]]   // correctly uses triplet[2]

  The tripletIndexes array stores {i, j, k} correctly at line 318.

NUANCED IMPACT:
The bug is ONLY in the eligibility filter that determines which triplets
are candidates for exclusion. The actual triplet exclusion logic works
correctly once a triplet is selected. This means:

1. Some valid triplets may be INCORRECTLY EXCLUDED from the candidate list
   (if middleOperator has many seats, double-counting makes it look like
   too many seats would be removed).

2. Some invalid triplets may be INCORRECTLY INCLUDED (if the k-th operator
   has many seats that would push the remaining count below the threshold,
   but this is not checked).

3. For case (2), the system would attempt to exclude a triplet that leaves
   too few participants, potentially causing DKG to fail even when excluding
   that triplet. This reduces the effective retry space.

4. For case (1), valid triplet exclusion candidates are skipped, meaning
   colluding triplets that SHOULD be excludable might never get excluded.

ATTACK SCENARIO:
If 3 malicious operators are colluding to disrupt DKG:
- Single operator exclusion (works correctly) will not catch them.
- Pair exclusion (works correctly) will not catch them if all 3 are needed.
- Triplet exclusion has a BROKEN eligibility filter, so the correct
  malicious triplet may not appear in the candidate list at all,
  or the candidate list may contain wrong triplets.

This means the retry mechanism may fail to ever try excluding the
correct malicious triplet, making DKG permanently stuck if 3 operators
collude. The claim of permanent DoS on wallet creation is plausible.

HOWEVER - SEVERITY MITIGATING FACTORS:
- Requires 3 colluding operators in the same DKG group (non-trivial).
- A new DKG attempt with different group composition would bypass this.
- The bug only affects the eligibility filter, not the exclusion itself.
- In practice, if all operators have equal seat counts, the bug may not
  manifest because double-counting j vs counting k gives the same result
  (assuming uniform seat distribution).

The bug is MOST impactful when operators have UNEQUAL seat counts
(i.e., some operators hold more seats than others).
