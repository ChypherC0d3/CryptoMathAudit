# Firedancer H6: AVX512 vs Reference Ed25519 Differential Test Plan

## What This Tests

Firedancer has TWO independent Ed25519 implementations:

1. **Reference** (`src/ballet/ed25519/ref/`): Uses fiat-crypto generated field
   arithmetic with 5x51-bit limbs (or 10x26-bit). Selected when `FD_HAS_AVX512=0`.

2. **AVX512** (`src/ballet/ed25519/avx512/`): Uses custom `fd_r43x6` representation
   with 6x43-bit limbs in 256-bit AVX-512 vectors. Selected when `FD_HAS_AVX512=1`.

If these produce different accept/reject results for ANY input, that is a
consensus-splitting bug. Validators running on AVX512 hardware (most datacenter
CPUs since 2017) would use the AVX512 path, while older hardware uses reference.

## Files

| File | Purpose |
|------|---------|
| `test_h6_avx512.sh` | Main build/run script |
| `test_h6_vectors.c` | Static test vectors (38+ from H1-H5 plus field edge cases) |
| `test_h6_inline_diff.c` | Field arithmetic + point decompression consistency tests |
| `test_h6_fuzz.c` | Long-running random fuzzer (sign/verify, field ops, sqrt_ratio) |

## Quick Start

```bash
# On an AVX512 Linux machine:
chmod +x test_h6_avx512.sh
./test_h6_avx512.sh

# If no AVX512 CPU, use Intel SDE emulator:
./test_h6_avx512.sh --sde

# Reference-only mode (for baseline on any machine):
./test_h6_avx512.sh --ref-only

# Extended fuzzing (100M iterations):
./test_h6_avx512.sh --fuzz-iters 100000000
```

## What Firedancer Already Has

Firedancer includes `fuzz_ed25519_sigverify_diff.c` which differentially tests
their Ed25519 against **ed25519-dalek** (Rust). This is NOT a ref-vs-AVX512
test; it compares Firedancer (whichever backend) against dalek.

Key point: their fuzzer always runs with the machine's native backend. On an
AVX512 machine, it tests AVX512-vs-dalek. On non-AVX512, it tests ref-vs-dalek.
There is NO existing test that directly compares ref-vs-AVX512.

## Divergence Risk Areas

### 1. Field Representation Differences

- **Reference**: `fiat_25519` with 5x51-bit limbs. Provably correct (generated by fiat-crypto).
- **AVX512**: `fd_r43x6` with 6x43-bit limbs. Hand-written AVX512 assembly.
  Multiple representation forms: arbitrary, signed, unsigned, unreduced, unpacked,
  nearly reduced, reduced.

Risk: The r43x6 carry propagation (fold_signed, fold_unsigned) could produce
different intermediate values that affect conditional branches.

### 2. is_zero Comparison Differences

- **Reference**: Checks if all limbs are 0 OR all limbs equal p's limb representation.
  Only checks TWO representations (0 and p).
- **AVX512**: Same approach but with different limb values for p in r43x6 form.

Risk: If an intermediate value reduces to a third representation of zero that
neither check catches, is_zero could return different results.

### 3. Point Equality (eq / eq_z1)

- **Reference**: Cross-multiplies X1*Z2 vs X2*Z1 and Y1*Z2 vs Y2*Z1, then uses
  fd_f25519_eq (which calls sub + is_zero).
- **AVX512**: Uses FD_R43X6_GE_IS_EQ which does the same cross-multiplication
  but in the QUAD representation, then checks via fd_r43x6_is_nonzero.

Risk: The AVX512 path uses `fd_r43x6_sub_fast` which produces signed
representations, then checks is_nonzero. If the sub_fast result is a non-zero
representation of zero that is_nonzero misses, the equality check diverges.

### 4. Point Decompression (sqrt_ratio)

Both use the same high-level algorithm (shared fd_f25519_sqrt_ratio in
fd_f25519.c), but the field operations dispatch to different backends.

Risk: The pow22523 chain involves ~250 squarings and multiplications.
Any carry propagation error accumulates through the chain.

### 5. Scalar Multiplication

Both share the same scalar mul code in fd_curve25519.c (wnaf-based), but the
point addition/doubling operations dispatch to different backends:
- Reference: fd_ed25519_point_add_with_opts -> fd_f25519_mul/add/sub
- AVX512: FD_R43X6_GE_ADD/DBL macros (operate on QUAD representations)

Risk: Different intermediate representations could cause different rounding
in the precomputed tables or wnaf evaluation.

### 6. Precomputed Tables

Both implementations use precomputed base point tables, but these are different
files: `fd_curve25519_table_ref.c` vs `fd_curve25519_table_avx512.c`.

Risk: If the tables were generated with different precision or rounding, the
scalar multiplication would diverge. (Low risk: tables are generated offline
and checked.)
